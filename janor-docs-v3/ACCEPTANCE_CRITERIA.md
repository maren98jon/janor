# Acceptance Criteria

## Purpose

This file defines the quality gate for MVP implementation. A feature is not done until these criteria are met.[cite:7][cite:32]

## Global acceptance criteria

- The app is clearly mobile-first in layout and behavior.[web:45]
- The UI feels premium, calm, and non-generic.[web:53][web:54]
- Core screens do not resemble an admin template or generic SaaS dashboard.[cite:32]
- All MVP screens support empty, loading, and error states.
- Realistic seed data is present.
- Core actions provide immediate user feedback.
- Copy sounds natural and product-real.
- Components look refined beyond library defaults.

## Home

Done when:
- Expiring soon section is visible and understandable at a glance.
- Cook today section is actionable.
- Quick add entry point is obvious.
- Layout remains elegant with zero, low, and high data density.

## Inventory

Done when:
- User can search inventory.
- User can filter by location and status.
- Item rows are easy to scan on mobile.
- Quick actions are available and understandable.

## Add item

Done when:
- New item can be added in under a minute without confusion.
- Form labels are clear.
- Validation is understandable.
- Success feedback is immediate.

## Recipes

Done when:
- Recipes can be browsed and filtered.
- Coverage or ingredient availability is understandable.
- Recipe detail supports cooking and planning.

## Weekly plan

Done when:
- User can assign recipes to days and slots.
- Weekly layout works on mobile.
- Shopping generation flow is understandable.

## Shopping

Done when:
- Active shopping list is easy to use one-handed.
- Checked and unchecked states are clear.
- Items can be added manually.
- Generated shopping logic feels believable.
